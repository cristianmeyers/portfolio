01 · contexte

Pourquoi migrer vers KEA ?

Dans le cadre de l'évolution du lab CyberEnv et de mon alternance à l'IUT Brest-Morlaix, j'ai réalisé la migration du service DHCP traditionnel vers la solution moderne ISC KEA.

isc-dhcp-server est en fin de vie active. ISC KEA le remplace avec une architecture modulaire, une API REST intégrée (Control Agent), une meilleure gestion des baux et une configuration JSON lisible.

Cette transition s'inscrit dans une démarche d'amélioration des performances, de la maintenabilité et de la scalabilité des services réseau du lab. Le résultat attendu est un service DHCP KEA pleinement opérationnel, avec gestion des pools d'adresses, réservations statiques, intégration VLANs et documentation complète.

Hostname siodhcp

Passerelle 192.168.10.254
DNS Primaire (AD) 192.168.10.4
Domaine sio.lan
Pool DHCP .50 → .150
Subnet 192.168.10.0/24
OS Ubuntu Server 22.04
03 · procédure de migration

Les 4 phases

De la préparation système à la supervision des baux — chaque phase est documentée avec les commandes et fichiers de configuration exacts.

1

Préparation du système

Définition de l'identité de la VM et configuration de l'interface réseau via Netplan.

hostname BASH 
$ sudo hostnamectl set-hostname siodhcp
$ exec bash
/etc/netplan/00-config.yaml YAML 
network:
  version: 2
  ethernets:
    ens18:
      addresses:
        - 192.168.10.2/24
      routes:
        - to: default
          via: 192.168.10.254
      nameservers:
        addresses: [192.168.10.4, 8.8.8.8]
        search: [sio.lan]
$ sudo netplan apply
2

Installation et authentification

Installation des paquets KEA. Lors de l'installation, une interface bleue demande de configurer l'authentification du Control Agent — choisir option 3 (configured_password).

installation BASH 
$ sudo apt update && sudo apt install -y kea-dhcp4-server kea-ctrl-agent

# Blue screen → option 3 : configured_password
# Mot de passe : sioPBA29200
3

Configuration des services

Deux fichiers JSON à configurer : le Control Agent (API sécurisée) et le moteur DHCPv4 (distribution des adresses).

/etc/kea/kea-ctrl-agent.conf JSON 
{
  "Control-agent": {
    "http-host": "127.0.0.1",
    "http-port": 8000,
    "authentication": {
      "type": "basic",
      "clients": [
        { "user": "kea-api", "password": "sioPBA29200" }
      ]
    },
    "control-sockets": {
      "dhcp4": {
        "socket-type": "unix",
        "socket-name": "/run/kea/kea-dhcp4-ctrl.sock"
      }
    }
  }
}
/etc/kea/kea-dhcp4.conf JSON 
{
  "Dhcp4": {
    "interfaces-config": { "interfaces": ["ens18"] },
    "lease-database": {
      "type": "memfile", "persist": true,
      "name": "/var/lib/kea/kea-leases4.csv",
      "lfc-interval": 3600
    },
    "subnet4": [
      {
        "id": 1,
        "subnet": "192.168.10.0/24",
        "pools": [{ "pool": "192.168.10.50 - 192.168.10.150" }],
        "option-data": [
          { "name": "routers",            "data": "192.168.10.1"  },
          { "name": "domain-name-servers", "data": "192.168.10.4"  },
          { "name": "domain-name",         "data": "sio.lan"       }
        ]
      }
    ],
    "loggers": [
      {
        "name": "kea-dhcp4",
        "output_options": [{ "output": "/var/log/kea-dhcp4.log" }],
        "severity": "INFO"
      }
    ]
  }
}
4

Activation et monitoring

Démarrage des services, activation au boot, puis surveillance des baux en temps réel.

démarrage des services BASH 
# Créer et attribuer le fichier de logs
$ sudo touch /var/log/kea-dhcp4.log
$ sudo chown _kea:_kea /var/log/kea-dhcp4.log

# Redémarrer et activer au boot
$ sudo systemctl restart kea-ctrl-agent kea-dhcp4-server
$ sudo systemctl enable kea-ctrl-agent kea-dhcp4-server
surveillance BASH 
# Logs en temps réel (ex: requêtes Windows)
$ sudo tail -f /var/log/kea-dhcp4.log

# Liste des baux actifs
$ cat /var/lib/kea/kea-leases4.csv
